# How the Event Loop Works

An interactive, F1-themed guide to the JavaScript event loop.

<ScrollStage>

<Section stage={1}>

## The Race Begins

Imagine an F1 race with a single rule: only one car can be on the track at a time. No overtaking, no side-by-side battles. Just one car, one track, one lap at a time.

That is JavaScript. It is a **single-threaded** language. It has one call stack, one thread of execution, and it runs your code line by line, in the order it appears. While it is doing one thing, it literally cannot do anything else.

This sounds limiting, right? And it is! If your one car stops in the middle of the track to change a tire, the entire race grinds to a halt. No rendering, no click handlers, no animations. The browser tab freezes. Users rage.

```js
// This blocks EVERYTHING for ~3 seconds
while (Date.now() < Date.now() + 3000) {
  // The car is stuck. Nothing else can run.
}
```

But here is the thing: this constraint is also a superpower. Because JavaScript only does one thing at a time, your code is **predictable**. You never have to worry about two functions fighting over the same variable, or a callback sneaking in mid-execution. The order is deterministic.

The question is: if we only have one car, how does JavaScript handle timers, network requests, and all the other async stuff the web demands? That is what the rest of this guide is about. Let us follow the car around the track.

</Section>

## The Call Stack

The call stack is our car's GPS. It tracks exactly where the car is at any moment: which function is running, and which functions are waiting for it to return.

Every time you call a function, it gets **pushed** onto the stack. When the function finishes and returns a value, it gets **popped** off. The car always executes whatever is sitting on top.

```js
function greet(name) {
  return `Hello, ${name}!`;
}

function welcome() {
  const message = greet("world");
  console.log(message);
}

welcome();
```

Walk through this like a pit crew radio call:

1. `welcome()` is called — pushed onto the stack
2. Inside `welcome`, we call `greet("world")` — pushed on top
3. `greet` returns `"Hello, world!"` — popped off
4. `console.log(message)` runs — pushed, then popped
5. `welcome` finishes — popped off
6. The stack is empty. The car has completed the lap.

The critical rule: the car **cannot move forward** until the current function on top of the stack finishes. If you call a function that calls a function that calls a function, they all pile up. The car does not come back to the main track until every nested call has returned.

This is synchronous execution. Predictable. Orderly. One thing at a time, like a car following the racing line through every turn. But what happens when our car needs something that takes a while, like fetching data from across the internet?

</Section>

## Web APIs

Here is where our F1 metaphor gets interesting. The car cannot leave the track, but the **team garage** has mechanics who can do work in the background.

When you call `setTimeout`, `fetch`, or add an event listener, you are not running JavaScript. You are sending a request to the browser's built-in features — the **Web APIs**. These are the garage crew. They work off-track, on their own threads, while the car keeps racing.

```js
console.log("Start");

setTimeout(() => {
  console.log("Timer done");
}, 1000);

console.log("End");

// Output: "Start", "End", "Timer done"
```

Here is what actually happens:

1. `console.log("Start")` runs immediately on the call stack
2. `setTimeout` is called — but it is a **facade function**. It does not do the timing in JavaScript. It hands the job to the browser's timer feature and immediately returns
3. `console.log("End")` runs while the browser counts down in the background
4. After 1000ms, the browser's timer finishes. But the callback does not jump straight onto the call stack...

This is the key insight Will Sentance drives home: `setTimeout` is not a JavaScript function in any meaningful sense. It is a label, a facade, for browser functionality that lives entirely outside the JavaScript engine. Same with `fetch` (network requests), `document` (the DOM), and even `console` (dev tools).

The JavaScript engine is just the car. The browser is the entire racing organization: the garage, the pit wall, the tire warmers, the telemetry. JavaScript gets to use all of it through these facade functions, but the work happens elsewhere.

So when the timer finishes, where does the callback go? It cannot just jump onto the call stack. That would break our "one car, one track" rule. It needs to wait its turn.

</Section>

## The Task Queue

When a timer finishes or a click happens, the browser does not slam the callback onto the call stack. That would be chaos — imagine a mechanic running onto the track mid-race. Instead, the callback goes into a **waiting area**: the task queue.

Think of it as a pit lane. Callbacks line up, single file, waiting for their chance to enter the track. And the event loop — our race marshal — enforces one strict rule: **one task per lap**.

```js
setTimeout(() => console.log("A"), 0);
setTimeout(() => console.log("B"), 0);
console.log("C");

// Output: "C", "A", "B"
```

Even with a delay of 0 milliseconds, those callbacks do not run immediately. Here is the full sequence:

1. Both `setTimeout` calls hand their callbacks to the browser's timer (0ms means "immediately complete")
2. Both callbacks enter the task queue — `A` first, then `B`
3. `console.log("C")` runs because it is synchronous, still on the call stack
4. All global code finishes. The call stack is empty.
5. The event loop checks: "Call stack empty? All global code done? Anything in the queue?" Yes! It grabs `A` and puts it on the stack
6. `A` finishes. The event loop goes around the loop again, then picks up `B`

This is the rule that trips everyone up: **the callback queue does not run until all global code has finished**. You could have a million `console.log` statements after `setTimeout(fn, 0)`, and every single one would execute before that callback gets its chance. The event loop is strict. Predictable. No exceptions.

As Will Sentance puts it: "You could have a million console logs, and still, everything would be left until after." That is not a bug. That is the design. Predictability over speed, every time.

</Section>

## The Microtask Queue

Now here is the plot twist. There is not just one pit lane. There are **two**. And one of them has VIP access.

When you use Promises — `.then()`, `.catch()`, or `async/await` — the callbacks do not go into the regular task queue. They go into a separate, higher-priority lane called the **microtask queue**.

The difference? The event loop **drains all microtasks** before it does anything else. Not one per lap. All of them.

```js
setTimeout(() => console.log("Task"), 0);

Promise.resolve().then(() => console.log("Microtask"));

console.log("Sync");

// Output: "Sync", "Microtask", "Task"
```

Even though `setTimeout` was called first, the Promise callback wins. Every time. Here is why:

1. `setTimeout` callback enters the **task queue**
2. `Promise.resolve().then()` callback enters the **microtask queue**
3. `console.log("Sync")` runs immediately (synchronous)
4. Global code is done. The event loop checks the microtask queue **first**
5. "Microtask" prints
6. Microtask queue is empty. Now the event loop checks the task queue
7. "Task" prints

This is what Jake Archibald emphasizes: microtasks run whenever the JavaScript stack empties. Not just between tasks — after every task, after every callback, after every event handler. The engine clears the entire microtask queue before moving on.

And here is the dangerous part: if a microtask queues another microtask, that one runs too, before any task or rendering gets a chance. Microtasks can **starve everything else**:

```js
// DON'T DO THIS — infinite microtasks block rendering forever
function forever() {
  Promise.resolve().then(forever);
}
forever();
```

The car never finishes the lap. The VIP pit lane has an infinite queue. The race marshal keeps sending cars through it, and the track never gets to repaint.

</Section>

## Rendering

After the car finishes its tasks and clears the microtask queue, the race marshal checks one more thing: does the track need to be repainted?

This is the **rendering step**. The browser may run these sub-steps:

1. **requestAnimationFrame** callbacks — your chance to update animations
2. **Style calculation** — computing which CSS rules apply
3. **Layout** — calculating where everything goes on the page
4. **Paint** — actually drawing pixels to the screen

The key word there is **may**. The browser is smart. If nothing visual has changed, it skips the entire rendering step. No need to repaint a track that looks the same. This typically runs at 60 times per second (every ~16.6ms), synced to your monitor's refresh rate.

This is why long-running tasks cause jank. If your JavaScript takes 200ms to run, the browser cannot render for that entire duration. The user sees a frozen screen, unresponsive buttons, choppy animations. The car is hogging the track and the paint crew cannot get on to refresh the lines.

```js
// This blocks rendering for ~2 seconds
for (let i = 0; i < 2_000_000_000; i++) {
  // Heavy computation — no rendering can happen
}
```

Jake Archibald makes an important point here: `requestAnimationFrame` is synchronized with the display's refresh rate, while `setTimeout` is not. A `setTimeout` loop might fire too often (wasting CPU) or not often enough (causing visual stutters). For animations, always use `requestAnimationFrame` — it is the browser telling you "now is a good time to update visuals."

The rendering step is the final piece of the lap. After it completes (or gets skipped), the event loop starts a brand new lap: check the task queue, drain microtasks, maybe render. Around and around, forever, until you close the tab.

</Section>

## Your Turn

You have seen the full lap now. One car, one track, three stops:

- **Task queue** — `setTimeout`, clicks, network callbacks. One per lap.
- **Microtask queue** — Promise `.then()`, `async/await`. Drained completely before moving on.
- **Rendering** — `requestAnimationFrame`, style, layout, paint. Skipped if nothing changed.

The interactive track above is your sandbox. Use the controls to add tasks and microtasks, then watch them flow through the queues and onto the call stack.

Here are some experiments to try:

- Add a `setTimeout(0)` and a Promise at the same time. Which runs first? (Spoiler: the Promise, always.)
- Queue several tasks and watch the event loop pick them up one per lap
- Notice how microtasks all run together before the next task gets a turn

One thing to keep an eye on: what would happen if microtasks kept spawning more microtasks, endlessly? The rendering step would never get a chance to run. The screen would freeze. This is called **rendering starvation**, and it is one of the most subtle bugs you can create with Promises. A future version of this visualization will let you see it happen in real time.

The event loop is not magic. It is a while loop with priorities. And now you know the order.
